# Многослойное туннелирование (Tunnel Chaining)

Туннели можно вкладывать друг в друга. Каждый слой добавляет свою функцию:
шифрование, маскировку, маршрутизацию. Это называется **chain** или **каскад**.

## 1. SOCKS5 over WebSocket over TLS (как в проекте)

```
Браузер → SOCKS5 (1080) → WS → WSS (443) → Caddy → NestJS → Цель
         └─── локально ──┘    └─── интернет ──┘    └─── серверно ──┘
```

**Слои:**
1. TLS — шифрование трафика
2. WebSocket — обход прокси-серверов (WS выглядит как обычный HTTP)
3. SOCKS5 — стандартный протокол для браузеров и приложений

Caddy в docker-compose делает WSS-терминацию: внешние клиенты
подключаются по WSS, Caddy расшифровывает и передаёт на NestJS по WS.

## 2. SSH в SOCKS5 через WebSocket

SSH уже умеет создавать SOCKS5-прокси (`-D`). Комбинируем:

```bash
# Шаг 1: SSH с SOCKS5 на локальном порту
ssh -D 1080 user@vps

# Шаг 2: направляем SOCKS5 через WebSocket тунель
# (используя local-proxy из проекта)
WS_URL=ws://hub.example.com:8080 node local-proxy/src/index.ts

# Шаг 3: браузер → SOCKS5 (1080) → SSH → VPS → интернет
#         браузер → SOCKS5 (1080) → WS → NestJS → VPS → интернет
```

В итоге трафик идёт через **два** туннеля одновременно.

## 3. WebSocket внутри SSH-туннеля

Обратная ситуация — SSH туннелирует WebSocket:

```bash
# SSH-туннель: локальный порт 9090 → сервер localhost:8080
ssh -L 9090:localhost:8080 user@vps

# Теперь WS-клиент подключается к localhost:9090
# (SSH перебросит на vps:localhost:8080)
```

Схема:

```
WS-клиент → SSH (9090) → VPS → NestJS (8080) → TCP → цель
```

## 4. Tor over WebSocket

Tor — анонимная сеть. Можно пустить трафик через WS-туннель в Tor:

```bash
# На сервере: запускаем Tor SOCKS5
# tor SocksPort 9050

# На клиенте: браузер → SOCKS5 → WS → Сервер → SOCKS5 → Tor → интернет
```

Схема:

```
Браузер → local-proxy → WS → NestJS → localhost:9050 → Tor → .onion
```

## 5. DNS over HTTPS WebSocket (маскировка)

DNS-запросы часто блокируют или перехватывают. Прячем DNS в WebSocket:

```
Клиент → DoH-резолвер → запрос HTTPS → WS-туннель → DNS-сервер → ответ
```

В проекте можно заменить SOCKS5 на DNS:

```
dgram (DNS-запрос) → WS → NestJS → UDP (8.8.8.8:53) → ответ
```

Уже есть UDP relay — это почти готовый DNS-туннель.

## 6. Практический пример: тройной туннель

Объединяем всё вместе:

```
# Терминал 1: SSH-туннель до VPS
ssh -L 8080:localhost:8080 user@vps

# Терминал 2: NestJS сервер на VPS (через SSH)
# Работает на порту 8080 (локально на VPS)

# Терминал 3: local-proxy подключается к localhost:8080
# WS идёт через SSH-туннель на VPS
WS_URL=ws://localhost:8080 node local-proxy/src/index.ts

# Терминал 4: браузер через SOCKS5 localhost:1080
# → local-proxy → WS → SSH → VPS → NestJS → интернет
```

## Сравнение схем

| Схема | Анонимность | Сложность | Скорость |
|---|---|---|---|
| Только WS-туннель | Низкая | Низкая | Высокая |
| WS + SSH | Средняя | Средняя | Средняя |
| WS + Tor | Высокая | Высокая | Низкая |
| WS + SSH + Tor | Очень высокая | Очень высокая | Очень низкая |

## Вывод

Туннели — это слои. Каждый слой решает свою задачу:

- **WebSocket** — обход L7-прокси и NAT, единое соединение
- **SOCKS5** — стандартный интерфейс для приложений
- **TLS** — шифрование и защита от обнаружения
- **SSH** — аутентификация и шифрование
- **Tor** — анонимность

Проект даёт базовый слой (WS). Ты можешь комбинировать его с любыми другими
технологиями для достижения нужного уровня безопасности и приватности.
